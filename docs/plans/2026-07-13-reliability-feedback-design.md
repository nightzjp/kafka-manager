# 稳定性与操作反馈设计

## 目标

让 Kafka Manager 在前端渲染异常、接口失败和危险操作场景下始终给出清晰、可恢复的反馈，避免整页白屏、静默成功和浏览器原生确认框带来的割裂体验。

## 设计原则

- 异常隔离：根组件之外增加异常边界。渲染错误时保留产品身份、错误摘要和“重新加载”恢复入口，不暴露调用栈。
- 反馈一致：成功、失败和提示统一进入右下角反馈中心；消息可被辅助技术读取，支持自动消失和手动关闭。
- 危险操作可核对：删除 Topic、重置 Consumer Offset 必须输入资源名称；配置回滚必须明确展示备份 ID 和影响。
- 防止重复提交：确认后进入等待状态，关闭和提交按钮同时锁定；失败后保留对话框并给出可继续操作的错误。
- 视觉连续：沿用现有深浅色变量、Lucide 风格图标和紧凑技术型界面，不引入新的 UI 框架。

## 组件与数据流

`main.tsx` 在 `App` 外依次挂载 `AppErrorBoundary` 和 `FeedbackProvider`。页面通过 `useFeedback()` 发送消息，Provider 管理消息队列和自动清理。

`ConfirmDialog` 是受控组件。业务页面保存待确认操作和表单快照；用户确认后由页面执行 API。这样确认框不包含 Kafka 业务知识，也不会把请求逻辑藏进通用组件。

## 错误处理

- 渲染异常：异常边界显示恢复页并记录到 `console.error`，用户可重新加载。
- API 异常：危险操作在对话框内显示错误，同时发送失败反馈；普通成功操作发送成功反馈。
- Clipboard 异常：复制名称会根据 Promise 结果显示成功或失败，不再静默。

## 测试

- 测试异常边界的派生状态与重试动作。
- 测试反馈队列的新增、上限和删除行为。
- 测试危险确认只有输入完全匹配时才能提交。
- 运行 Vitest、TypeScript、Vite build、Go test 和 vet，确保嵌入式前端与后端均未回归。
