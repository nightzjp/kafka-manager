# Kafka 消息过滤设计

## 目标

面向研发排查场景，为普通消息查询和实时跟随增加一致的 Key、Value 文本及 JSON 字段过滤能力，并明确区分扫描量和返回量。

## 过滤能力

- Key：精确匹配、包含、前缀匹配。
- Value 文本：精确匹配、包含。
- JSON：最多 5 个条件，条件之间使用 AND。
- JSON 路径：点号访问对象，数字段访问数组，例如 `data.user.id`、`items.0.sku`。
- JSON 运算符：等于、不等于、包含、存在、大于、大于等于、小于、小于等于。

等值比较尊重 JSON 原始类型：数字按数字比较，布尔值按布尔比较，字符串按文本比较。路径不存在时条件不匹配；`exists` 只判断路径是否存在。

## 扫描模型

Kafka Broker 不支持按消息内容查询。服务从用户指定的起始位置消费消息，最多扫描 `scanLimit` 条，并在服务端逐条过滤；匹配结果达到 `limit` 后停止。

- 默认扫描 5,000 条，最大 50,000 条。
- 默认返回 100 条，最大 500 条。
- 没有任何内容过滤时，扫描量自动等于返回量，保持当前查询性能。
- 响应返回扫描数量、匹配数量、JSON 解析失败数量和是否达到扫描/结果上限。

过滤在消费批次内完成，不把 50,000 条原始消息全部保存在内存中。

## API

查询参数增加：`keyFilter`、`keyOperator`、`valueFilter`、`valueOperator`、`jsonFilters`、`scanLimit`。`jsonFilters` 是 URL 编码后的 JSON 数组。

实时 SSE 使用同一组过滤参数。无效运算符、空 JSON 路径或超过 5 个条件返回 400，不静默忽略。

## 前端交互

基础查询栏保持紧凑，新增“高级过滤”展开区。过滤条件摘要显示启用数量；结果区显示“已扫描 N 条，匹配 M 条”，JSON 解析失败时单独提示。多个 JSON 条件明确标注为 AND。

## 验证

- Go 单元测试覆盖字符串、嵌套对象、数组、类型比较、无效 JSON 和限制校验。
- TypeScript 单元测试覆盖参数序列化和空条件剔除。
- 全量前后端测试、类型检查、生产构建、Go vet 和真实运行接口验证。
